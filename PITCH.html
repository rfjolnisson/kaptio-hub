Initial Assessment

Decap and Keystatic lack built-in approval workflows entirely. Tina has excellent UX but no approval features. These three would all require building custom approval systems on top, which defeats the purpose of using an existing CMS.

This narrows our exploration to two viable approaches: CloudCannon (mature commercial platform) or Custom Build (purpose-built solution). Halldor will explore both during the POC phase.

CloudCannon Path

Content Workflow: How Could It Work?

If we choose CloudCannon, here's how the end-to-end content creation process could work:

1

Request

Any team requests content via backlog: "Need Bunnik program setup guide" or "Need Connect API one-pager"

2

Generate

Kai (via Cursor) creates markdown file in CloudCannon format. Commits to Git branch. First draft ready.

3

Review + Edit

Content team opens in CloudCannon's visual editor. Reviews accuracy, polishes messaging, adds media. Can use Notion AI.

4

Approve

Submitted via CloudCannon workflow. Product + Marketing teams approve. Merges to main.

5

Publish + Track

Content live on hub. Teams share with appropriate audience (prospects, customers, partners). Analytics captured.

Key Point: Kai's Role

Kai is a content generation tool, not an embedded assistant. Team members use Kai via Cursor to create first drafts. Kai never interacts with customers or prospects. This sidesteps all security/privacy concerns while still leveraging AI for content creation speed.

With CloudCannon

  • ‚Ä¢ Team learns CloudCannon's patterns
  • ‚Ä¢ Relies on their documentation
  • ‚Ä¢ Limited by their feature set

With Custom Build

Both paths can work. Both have trade-offs. The POCs will give us the data to make this choice intelligently.
